The action button component creates a large button with an icon, header, and details. These buttons are used in sets to present users with multiple, branching options to move forward with a task.

The sky-action-button-container component insures that all inner sky-action-button coponents have consistent height and spacing. The sky-action-button component consists of three inner components: sky-action-button-icon, sky-action-button-header, and sky-action-button-details.

Fires when the user clicks the action button. Specifies an icon to be used from the Font Awesome library. For example, to use the fa-filter icon, use filter as the iconType.
Usage

Use action buttons when:

  • The user has initiated a task or workflow
    • This is often done via a page's primary action
  • The task has between 2 and 6 methods of completion
    • The user must choose exactly one option
    • If there are more relevant choices, consider a different selection input and interaction
  • The choices are system-defined, not user-defined
  • The choice of method will determine how the majority of the workflow will proceed
    • This choice is made at or near the beginning of the task
  • The outcome of end result will be largely the same, regardless of the method

Action buttons must represent a necessary step in any workflow they inhabit - they should not be used in ways where they can be skipped, avoided, or answered later. Present them in a way that isolates the choice they represent and removes other decisions, different tasks, or other content. Take advantage of the "details" line in the button to give users context or explain the differences between the options.

When to use action buttons

Use action buttons immediately after a user has initiated a task, to differentiate between different branches the task may follow.

Use action buttons on a page or in a dialogue, but only when a task is underway and the choice about how to complete the task is the only choice relevant to the user. Do not display any other inputs or choices until the user picks an option. When the user picks an option, hide the action buttons (or close the modal).

When not to use action buttons

Do not use a single action button by itself. Action buttons should not serve as a primary call-to-action in a page where many other actions are available to the user.

Do not use actions buttons to present different methods for completing a task when the user has not initated the task yet. Action buttons should not compete against other content or actions for the userâ€™s attention.

Do not use actions buttons when the consequential differences between the choices are a small minority of the scale of the task, or when the choice would not be made at the beginning of the task. Use a different selection input and progressive disclosure to reveal additional content, if needed.

Workflow placement and behaviors

Use a set of action buttons inside a modal when no page already exists, or the remaining bulk of the task after the action button decision takes place inside a modal. This is common when creating a new record that only requires a standard modal during different versions of its 'Add' process.

  • Use a 'default' size modal for 2 action buttons and a 'largeModal' for 3 or more.
    • Center the action buttons inside the modal, wrapping to another row if needed.
  • When the user selects a choice, dismiss this modal and open whatever container is appropriate for the remainder of the task.
    • If the user cancels or abandons the task afterwards, do not return the user to this modal.
  • Always provide a "Cancel" action inside a modal with action buttons.

When the user is completing an Add or configuration process contained inside a page or tab (that includes no other task), use action buttons directly on the page. Most 'Add' processes will use the modal example above.

  • If necessary, include a question above the action buttons to explain the decision being made.
  • Center the question and action buttons inside the page.
  • When the user selects a choice, remove the action buttons and replace them with the consequent next steps of the workflow.
  • If the user leaves the page without selecting a choice, the record or process should be treated as "incomplete" and have any appropriate status indicators on its parent view.
    • An explicit "Cancel" or "Delete" action underneath the action buttons and be used if it is valuable for the user to delete the record entirely from this point.

When the user is adding a new user-defined tab and choosing how populate its content, use action buttons directly inside the tab, rather than inside a modal.

  • There is no need for a "Cancel" action underneath the action buttons, since user-defined tabs can be removed on the tab control.
